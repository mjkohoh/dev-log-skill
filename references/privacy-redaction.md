# 隐私与脱敏规则

开发日志默认可以提交到 git，因此必须默认脱敏。宁可少写，也不要把敏感数据写入仓库。

交接快照默认不提交到 git，但仍必须默认脱敏。它通常更接近当前工作台状态，可能包含临时凭证、原始响应、局部配置或未清理的调试信息，因此不能因为“临时”就放松规则。

## 禁止原样记录

禁止原样记录：

- API key、access token、refresh token、cookie、session、JWT。
- 密码、私钥、证书内容、SSH key。
- 真实手机号、身份证号、银行卡号、详细住址。
- 生产用户原始数据、客服对话、订单明细、医疗或财务敏感信息。
- 带签名参数、授权参数、临时凭证的 URL。
- `.env`、生产配置、数据库连接串中的敏感值。
- 内网服务密钥、Webhook secret、第三方平台 secret。

## 推荐脱敏写法

可以记录事实，不记录值：

```md
- 使用了 `DASHSCOPE_API_KEY` 环境变量，但未记录具体值。
- 生产 API 返回 401，确认是 token 过期导致。
- 数据库连接串已脱敏为 `postgres://user:***@host/db`。
- 测试账号邮箱已脱敏为 `u***@example.com`。
```

## URL 脱敏

带敏感参数的 URL 不要原样记录。改写为：

```md
- 请求 `https://api.example.com/v1/orders?...` 返回 403，签名参数已省略。
```

## 错误输出脱敏

命令输出、堆栈、HTTP 响应中可能夹带敏感数据。写日志前先检查：

- 是否包含 token、key、secret、password。
- 是否包含真实用户数据。
- 是否包含内部主机、账号或连接串。

只记录必要结论，不粘贴大段原始输出。

## 无法确认时

如果 agent 无法判断内容是否可以安全提交：

1. 不写入可提交日志。
2. 不写入交接快照的敏感原文。
3. 向用户说明风险。
4. 建议写入本地私有笔记，或等待用户确认。

## 纠错和泄露处理

如果发现旧日志已经写入敏感信息：

- 不要在新日志里重复敏感值。
- 提醒用户需要清理历史记录。
- 如需记录处理过程，只写脱敏描述。
