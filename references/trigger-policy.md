# 触发策略

开发日志应当主动但克制。它记录关键上下文，不记录聊天流水账。

## 策略档位

| 策略 | 行为 |
|---|---|
| `explicit_only` | 只有用户明确要求时才记录 |
| `conservative` | 默认策略，只在关键节点自动记录 |
| `balanced` | 有明显阶段性开发进展时也记录 |
| `eager` | 更频繁记录，适合长任务、调试、研究型开发 |

交接快照不使用这些自动记录档位作为主要触发依据。它默认由用户明确要求触发，例如切换 agent、生成 handoff、暂停并保存当前状态。

## `conservative` 默认记录场景

在默认策略下，这些场景应记录：

- 用户明确要求记录日志、写 dev log、保存开发过程。
- 做出重要技术决策或改变实现方案。
- 完成跨多个文件或模块的一组改动。
- 遇到阻塞并定位到原因。
- 测试失败后找到根因。
- 任务完成并有明确验证结果。
- 准备暂停、切换任务，或上下文可能中断。

其中“切换任务或上下文可能中断”通常写 dev log；如果用户明确说要切换到另一个 agent 或生成当前状态文档，应写交接快照。

## 默认不记录场景

这些场景通常不记录：

- 简单问答。
- 单行小改。
- 纯格式化。
- 只读代码但没有形成可复用结论。
- 常规命令正常执行，且没有产生新决策、新风险或新事实。
- 临时失败但没有定位结论。

## 用户主动要求记录

用户明确要求时，agent 应默认尝试记录，但仍要执行：

1. 去重检查。
2. 价值判断。
3. 敏感信息脱敏。
4. 必要时追加 update，而不是重复写完整条目。

如果已有等价记录，简短告知用户，无需重复追加。

## 用户主动要求交接

用户明确要求切换 agent、生成 handoff、整理当前开发状态时，agent 应生成或更新交接快照，而不是追加普通开发日志。

默认行为：

1. 读取当前目标、用户约束、git diff 摘要和相关 dev log。
2. 覆盖 `.agents/handoff/current.md`。
3. 只保留下一个 agent 接手需要的信息。
4. 完成脱敏检查。
5. 告诉用户交接文件路径。

如果用户要求保留历史，再额外生成带时间戳的 handoff 文件。

## 价值判断

自动记录前问一个问题：

> 这条日志能否帮助未来的人或 agent 节省大约 5 分钟以上？

如果答案是否定的，不要自动记录。用户明确要求时可以记录更小的信息，但仍要避免重复和敏感内容。
